# KU 文档写作经验

本目录沉淀“怎么写好如流知识库文档”的经验，尤其是产品需求、计费策略、方案对比、讨论稿这类文档。这里记录的是表达方式、结构选择和迭代习惯，不记录某次业务的具体数值、具体决策、账号配置或工具参数。

## 使用时机

在以下场景开始前先读本文件：

- 用户要求写、改、整理、精简 KU 知识库文档。
- 用户给出一堆口述信息，需要整理成需求、方案或讨论稿。
- 用户连续反馈“太长”“太口语”“太死板”“不用为了写而写”等写作风格问题。
- 文档需要在“已有结论、待讨论事项、方案对比、上线后观察”之间做取舍。

在文档写完或用户反馈后，如果学到可复用的表达习惯，需要把经验补充回本目录。

## 核心原则

### 只写能帮助决策的信息

不要为了像 PRD 而堆章节。若信息量不多，可以只保留背景、目标、方案、待讨论。竞品、流程、指标、风险等章节只有在确实支持判断时才写。

判断一个小节是否保留时，问三个问题：

- 读者看完是否能更快做决策？
- 这部分是否来自已知信息，而不是为了完整性补出来？
- 如果删掉，文档结论是否会变弱？

### 分清“已确定”和“待讨论”

用户明确给出的方案、约束、口径，写入正文方案区。还没有定的内容放入“待讨论”，不要替用户过度拍板。

待讨论不是垃圾桶，只放会影响产品、销售、研发或运营选择的问题。细枝末节先不写。

### 结论先行，但结论要像判断

背景、目标、风险这类分析性内容，优先用：

```text
1、**观点句**：展开说明。
2、**观点句**：展开说明。
```

加粗部分不要写成长段，也不要机械压成四字词。它应该是一个读者能直接理解的判断，例如：

- 好：**按月清零会浪费低峰、卡住高峰**
- 好：**年度权益要清晰**
- 好：**高频风险要可控**
- 不好：**周期波动**，过短，像标签，不像判断。
- 不好：**管理员要能快速救急**，太口语。
- 不好：**为了保证用户权益和管理员效率，需要建立一套可审计的动态额度调控机制**，太长，读者抓不住结论。

展开句只解释结论，不另起新观点。

### 表格列数跟问题复杂度匹配

表格不是越全越好。先判断这个表解决什么问题，再决定列。

常见拆法：

- 对比多个价格档位：`方案 / 价格 / 权益 / 定位 / 备注`。
- 对比多个计费或额度策略：`方案 / 初始状态 / 触发后处理`。如果管理员能力、风险说明只是补充，不一定单独成列。
- 解释统计口径：单独用 `分位点 / 数值 / 说明`，不要把口径塞进长段落。
- 展示推算结果：单独用 `方案 / 关键权益 / 按不同假设可覆盖情况`。

如果用户要求“改为三列”，通常是在提醒表格承担的决策问题更简单，应把辅助说明合并到单元格里，而不是保留过多列。

### 先定义口径，再放推算

涉及分位数、日均值、覆盖天数、渗透率、风险阈值时，先用一张小表解释口径，再用另一张表展示推算结果。

不要把“数据从哪里来、代表什么、怎么算出来”写成一长段。表格更适合承载这种信息。

### 用词保持产品文档感

表达要直接，但不能过度口语。目标是“像产品经理给团队看的判断”，不是营销文案，也不是聊天口吻。

推荐：

- “权益清晰”
- “价格策略要有弹性”
- “高频风险要可控”
- “管理员干预要高效”

避免：

- “买到了什么”
- “快速救急”
- “别人权益”
- “卡死用户”

如果用户说“太通俗”，优先把口语词换成产品词；如果用户说“太死板”，优先把抽象标签换成判断句。

## 自进化规则

每次完成 KU 文档写作或修改后，做一次 1 分钟复盘：

1. 用户这次纠正的是结构、语气、表格、范围还是判断方式？
2. 这个纠正是否可复用到未来 KU 文档？
3. 如果可复用，更新本文件；如果有完整迭代链路，新增一个 `examples/*.md` 案例。
4. 案例记录“问题 -> 用户反馈 -> 调整方式 -> 可复用规则”，不要记录敏感信息、账号、具体业务数值或一次性配置。

经验沉淀要克制。不是每个改字都值得记录，只有能改变下一次写作方式的经验才写。
