# 案例：高校版计费策略文档迭代

这个案例记录一次 KU 需求文档从“信息整理”到“表达打磨”的迭代过程。案例只沉淀写作方法，不沉淀具体业务数值和最终方案。

## 初始问题

用户提供的是一组待讨论的定价档位、年度额度思路、触顶后的额度处理方式，以及对高频用户风险的担忧。文档目标不是完整 PRD，而是把“已确定的方案”和“仍待讨论的问题”说清楚。

## 迭代过程

### 不要为了写而写

最初容易按完整 PRD 展开，写出较多章节。用户反馈信息不多，不需要硬凑。

可复用规则：

- 信息少时，优先保留必要结构：背景、目标、方案、待讨论。
- 观察指标、MVP 建议、待确认等如果没有足够信息，可以合并成一个“待讨论”。
- 文档的价值是帮助决策，不是看起来完整。

### 表格要跟决策问题对齐

计费方案最初拆了较多列，包含管理员能力、说明、风险等。用户希望改成三列。

可复用规则：

- 当核心问题是“不同方案怎么发、触顶后怎么办”，三列足够：`方案 / 初始发放积分 / 触顶后方案`。
- 解释性内容可以合并到“触顶后方案”里，不必单独成列。
- 列数越多，读者越容易把注意力放到细节，而不是关键取舍。

### 统计口径适合单独成表

用户希望把 P25、P50、P75 这类分位点也表达为表格。

可复用规则：

- 先用一张表说明口径：`分位点 / 日均消耗 / 说明`。
- 再用一张表展示推算：`方案 / 年度权益 / 按各分位点可覆盖情况`。
- 不要把分位点口径和推算结果揉在一段话里。

### 结论先行不是机械标题

背景和目标一开始是普通段落，用户希望按金字塔原则写：先抛观点，再展开。后来又指出加粗结论不能太长，也不能机械压成四字词。

可复用规则：

- 分析性段落用 `1、**观点句**：说明`。
- 加粗部分要像一个判断，读者只看加粗也能理解逻辑。
- 不要把观点压成抽象标签，也不要写成完整长句。

### 语气要在“通俗”和“专业”之间校准

一次调整中过于口语化，例如把风险写成“不能无感挤占别人权益”，把管理员能力写成“快速救急”。用户反馈这类表达太口语。

可复用规则：

- 面向产品/研发/销售协作的 KU 文档，表达要直接，但保留产品文档感。
- “高频风险要可控”比“不能无感挤占别人权益”更合适。
- “管理员干预要高效”比“管理员要能快速救急”更合适。
- 用户说“太死板”时，改成判断句；用户说“太口语”时，换成产品词。

## 本案例沉淀出的检查清单

写完类似 KU 文档后，快速检查：

1. 是否只写了当前决策需要的信息？
2. 是否把已确定方案和待讨论问题分开？
3. 分析性内容是否结论先行？
4. 加粗结论是否是清楚的判断，而不是标签或口语？
5. 表格列数是否服务当前问题，而不是追求完整？
6. 涉及数据口径时，是否先解释口径再展示推算？
